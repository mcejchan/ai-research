## Verified findings

### 1. The quiz retains an obsolete, divergent second runtime authority

**Problem.** `quiz/server.js` still discovers and projects levels through `/api/levels` and serves `/api/level`, duplicating the static manifest/data path used by the actual application.

**Evidence.** The browser exclusively fetches `levels/index.json` and `levels/<path>` (`quiz/app.js`); `.github/workflows/deploy-quiz.yml` builds that manifest and deploys `quiz/` as a static Pages artifact. A maintained-code search found no consumer of `/api/levels`, `/api/level`, or `quiz/server.js`. Git commit `8b90375` explicitly migrated the browser from both API calls to static files but retained the server. The retained implementation has already drifted: it scans `index.json` as a level and recognizes only `easy`; against current data it projects 21 rows for 20 source levels and maps all four `extra-easy` levels to `hard`. The canonical generated manifest has exactly 20 paths and preserves four `extra-easy` values.

**Impact.** Two implementations claim level-discovery and metadata ownership. One is dead from the deployed flow yet remains plausible to developers and stale documentation, can report incorrect data, and doubles every metadata/schema change. It is complexity with negative compatibility value.

**Recommendation.** Delete `quiz/server.js` and treat source level JSON → `build-index.js` → static browser reads as the sole quiz path. If local serving is still useful, use a generic static server command rather than a quiz-specific API. Before deletion, a bounded external-caller check is sufficient; no in-repository caller exists.

### 2. Test ownership is fragmented and the declared gate does not represent the repository

**Problem.** The registry's only project test authority is `cd .../youtube-transcript-pipeline && pytest`, while maintained executable tests also exist for quiz and yt-viewer; deployment validates none of them.

**Evidence.** `quiz/build-index.test.js` and `yt-viewer/server.test.js` are maintained tests but are absent from the registry command. Running both showed the viewer test passing and the quiz test failing: after commit `a578605` changed difficulty handling from a closed `easy|hard` normalization to `data.difficulty || 'hard'` for `extra-easy`, the test still requires unknown `weird` to normalize to `hard`. The declared Python command is also not reproducible in this environment: `pytest` is absent from PATH; `python3 -m pytest` reaches collection but fails because `DRIVE_FOLDER_ID` is read unconditionally at module import. The Pages workflow builds and uploads without a test step; recent green deploys include the commit that introduced the quiz contract/test divergence.

**Impact.** “Tests pass” has no single project meaning. Depending on entrypoint, two applications are silently ignored, the selected suite can fail before collection due to operator configuration, and deployment green does not imply even its own builder test is green. This increases cognitive load and permits drift precisely at subsystem boundaries.

**Recommendation.** Establish one small project-level gate (script or Make target) that invokes the existing Node suites plus pipeline pytest with explicit test-safe configuration. Point the registry at that gate and make Pages run at least the quiz slice—or preferably the same gate—before upload. This centralizes orchestration, not application architecture; keep the three subsystem tests separate underneath. Resolve the difficulty contract explicitly: preserve the known enum (`extra-easy|easy|hard`) while rejecting/defaulting unknown values, then align the test.

### 3. Tracked `tmp/` is stale workflow residue and preserves superseded architecture

**Problem.** Five committed files under `tmp/` are task/learning staging artifacts rather than project inputs or outputs.

**Evidence.** Maintained-code searches found no reader. Their history ends in May 2026. `tmp/content.md` still instructs contributors that the browser renders from `/api/levels`, directly contradicting the static app since commit `8b90375`. Equivalent canonical learning material exists under `learnings/`; `.gitignore` does not ignore `tmp/`.

**Impact.** Search results surface obsolete architectural advice, duplicate canonical learnings, and make disposable agent artifacts look repository-owned.

**Recommendation.** Remove tracked `tmp/` contents and ignore the staging directory (or move genuinely durable text into its existing canonical `learnings/` destination). Do not create a new archival mechanism; Git history already preserves the old files.

## Suspicions

None left open. External use of the quiz API cannot be proven impossible from repository evidence, so the recommendation includes a bounded caller check rather than treating absence as metaphysical proof.

## Rejected hypotheses

- **Static manifest versus source level JSON is a duplicate authority.** Rejected: source JSON owns content; the manifest is a deterministic deployment index. Current paths match 20/20, and removing the manifest would reintroduce runtime discovery unavailable on static Pages.
- **The yt-viewer server should be merged with quiz serving.** Rejected: yt-viewer exposes a live content API and has a passing test; quiz production is static. Merging would broaden coupling and deployment blast radius.
- **Transcript API → Whisper and Drive → local fallback should be collapsed.** Rejected: evidence shows capability and offline/storage boundaries, not two authorities for the same successful state. This review did not establish redundancy or harmful cascading behavior.
- **Commit the generated manifest less often / delete it.** Not recommended without changing content authoring and Pages guarantees. The current generated file matches all source paths, and the build regenerates it deterministically.

## Self-critique

The strongest simplification is deletion, but only for the quiz server whose consumers disappeared in the static migration and whose output demonstrably diverges. The proposed unified gate does not merge application internals or force one framework; it centralizes only the definition of repository health. Adding orchestration has a small maintenance cost, but it replaces an already-declared misleading command and a no-test deployment path rather than adding another authority. `tmp/` cleanup removes duplication rather than relocating it because durable learnings already have a canonical directory.
